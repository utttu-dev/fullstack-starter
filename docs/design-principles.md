# 設計思想

このテンプレートが大事にしている考え方。

---

## 環境構築だけでなく、QCDを最大化する

スキャフォールディングツールの多くは「環境を作る」ことだけを目的にしている。
このテンプレートは、それに加えて開発の Quality(品質) / Cost(コスト) / Delivery(スピード)
を高める仕組みを最初から組み込むことを目指している。

```
Quality(品質)
  - CIでのLint/Test/Build/E2Eの自動チェック
  - readinessProbeによるヘルスチェック

Cost(コスト)
  - マルチステージビルドによるイメージの軽量化
  - 依存関係キャッシュを活かしたビルド高速化

Delivery(スピード)
  - ローカル → コンテナ → Kubernetes の段階的な検証フロー
  - 1コマンドで完結する開発フロースクリプト
```

---

## アプリケーションコードを含めない

このテンプレートは「型」だけを提供し、ビジネスロジックは一切含まない。

```
理由
─────────────────────────────
アプリケーションコードを含めると
それ自体が「もう1つの固有実装」になってしまい
テンプレートとしての中立性が失われる

→ どんな言語・フレームワークの組み合わせでも
  同じ構成パターンを適用できる状態を保つ
```

---

## ローカル → コンテナ → 本番 の3段階で検証する

```
① ローカル
   最速でバグを検知する(ホットリロード等)

② コンテナ(Docker Compose)
   環境差異によるバグを検知する

③ 本番相当(Kubernetes)
   スケーラビリティ・可用性を検証する
```

それぞれの段階で同じアプリケーションが動くことを確認しながら開発を進める。

---

## 設定は環境変数で外部化する

データベース接続情報やCORS許可オリジンなど、環境によって変わる値はすべて
環境変数(`.env` / ConfigMap / Secret)で管理し、コードに直接書き込まない。

```
理由
─────────────────────────────
同じイメージを開発・本番で使い回せるようにするため
設定変更のたびにビルドし直す必要をなくすため
```

---

## 既存ツールとの違い

```
create-react-app 等
  → 環境構築のみ。CI/CD・k8sは含まれない

Backstage
  → 大規模組織向けで構成が複雑

Cookiecutter
  → テンプレート生成のみ。QCDへの配慮がない
```

このテンプレートは、個人〜小規模チームでも導入しやすい軽量さを保ちながら、
Docker・CI/CD・Kubernetesという「本番運用を見据えた構成」を最初から
持っている点を特徴としている。
